# Contributing

Thanks for considering a contribution to `claude-hooks-pack`.

This pack ships user-level [Claude Code](https://docs.claude.com/en/docs/claude-code)
hooks — small shell / Python scripts that fire on lifecycle events
(`SessionStart`, `SubagentStart`, `UserPromptSubmit`, `PreToolUse`,
`Stop`) to nudge or block the agent's behavior. They live at
`~/.claude/hooks/` on every developer machine that runs Claude Code.

New hooks must pass the design rubric in [`docs/HOOK_DESIGN_RUBRIC.md`](docs/HOOK_DESIGN_RUBRIC.md)
§8 and ship all 8 self-check items below.

---

## Local development

```bash
git clone https://github.com/liush2yuxjtu/claude-hooks-pack.git
cd claude-hooks-pack

# (One-time) install pre-commit + shellcheck + shfmt
brew install shellcheck shfmt pre-commit   # macOS
pre-commit install

# Run the existing test to confirm a clean baseline
bash test/straight-fix-no-ask.test.sh

# Make your change.
# Before opening a PR, run:
pre-commit run --all-files
bash test/straight-fix-no-ask.test.sh
```

CI enforces the same checks via `.github/workflows/ci.yml`.

## New-hook checklist (rubric §8)

A new `<name>.sh` (or `.py`) is acceptable when it satisfies **all 8** items:

- [ ] **1. Trigger list** ≥ 100 OR ≤ 100 with a header comment explaining why
- [ ] **2. Structural trigger** present (slash command, XML marker, or compound regex)
- [ ] **3. Test harness** `test/<name>.test.sh` with ≥ 3 positive + ≥ 3 negative
- [ ] **4. Default silent-approve**; on hit, inject (not block)
- [ ] **5. Escape hatch** `CLAUDE_<NAME>_DISABLED=1` short-circuits the hook
- [ ] **6. VALUE contract** header references `~/.claude/CLAUDE.md` VALUE cascade (root → L1-L4 → LEAVES.md)
- [ ] **7. Logs** audit JSONL at `~/.claude/hooks/logs/<name>-audit.jsonl` + per-session `<name>-<session_id>.md` paths documented
- [ ] **8. Wiring** listed in `settings/hooks.fragment.json` under the correct lifecycle event

When in doubt, copy the structure of an existing hook that already passes
(e.g. `hooks/value-guard.sh`, `hooks/pair-chrome-soft-gate.sh`).

## One-off contribs

If your addition is project- or incident-specific (won't generalize to
other Claude Code users), put it under `contrib/one-offs/<name>/` with
a `TOP-NOTE.md` explaining:
- when the incident happened
- which project / machine it targeted
- what symptom it fixes
- what it's NOT meant to be (don't lie about reusability)

`install.sh` does **not** install contrib one-offs by default.

---

## Replacing an archived hook

If your PR reactivates a hook from `hooks/_archive/learned-mistakes/`:

1. Read the `INDEX.md` entry first — understand the original incident it
   replaced.
2. Move the script back to `hooks/`.
3. Add the appropriate entry in `settings/hooks.fragment.json`.
4. Update the rubric compliance vs the new version.

---

## PR conventions

- **One concern per PR.** Don't mix new-hook adds with refactors.
- **Reference the rubric §8 item numbers** your hook satisfies in the PR
  description.
- **Reference the archive INDEX entry** if you're reactivating an archived
  hook.
- **CI must pass.** Run `pre-commit run --all-files` locally first.
- **Don't touch** `settings/hooks.fragment.json` in a hook-PR unless the
  hook is genuinely new and needs wiring. The fragment is generated by
  `bin/build-fragment.sh` on the source machine, not by hand.
